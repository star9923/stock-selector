# 板块数据缓存优化说明

## 优化策略

采用**分层缓存**策略，将板块数据分为两部分：

### 1. 板块列表（长期缓存）
- **内容**：板块名称、板块代码
- **缓存时间**：30天
- **数据源**：同花顺（稳定）或东方财富
- **缓存文件**：`.cache/board_list_cache.json`

### 2. 板块行情（短期缓存）
- **内容**：涨跌幅、上涨下跌家数、情绪得分
- **缓存时间**：24小时（实时获取时1小时）
- **数据源**：仅东方财富
- **缓存文件**：`.cache/board_quote_cache.json`

## 工作流程

```
板块热度请求
    ↓
┌─────────────────────┐
│  获取板块列表       │
│  (30天缓存)         │
│  ✅ 同花顺稳定可用  │
└─────────────────────┘
    ↓
┌─────────────────────┐
│  获取板块行情       │
│  (24小时缓存)       │
│  ⚠️ 东方财富        │
└─────────────────────┘
    ↓
┌─────────────────────┐
│  合并数据返回       │
└─────────────────────┘
```

## 优势

### 1. 提高稳定性
- **板块列表**：从同花顺获取，非交易时间也可用
- **长期缓存**：板块列表不常变，30天缓存足够

### 2. 减少API调用
- **板块列表**：30天内只需获取一次
- **板块行情**：失败时使用24小时内的缓存

### 3. 降级优雅
- **完全离线**：使用缓存数据
- **部分失败**：板块列表可用，行情使用缓存

## 使用场景

### 场景1：首次使用（无缓存）
```
1. 从同花顺获取板块列表 → 缓存30天
2. 从东方财富获取板块行情 → 失败（非交易时间）
3. 结果：显示板块列表，但无行情数据
```

### 场景2：交易时间更新
```
1. 使用板块列表缓存（30天内有效）
2. 从东方财富获取板块行情 → 成功 → 缓存24小时
3. 结果：显示完整数据
```

### 场景3：非交易时间（有缓存）
```
1. 使用板块列表缓存
2. 从东方财富获取板块行情 → 失败
3. 使用板块行情缓存（24小时内）
4. 结果：显示缓存数据
```

## 更新缓存

### 自动更新
- 板块列表：30天后自动过期，重新获取
- 板块行情：每次成功获取时自动更新

### 手动更新
```bash
# 在交易时间运行
python3 update_board_cache.py
```

这会：
1. 强制更新板块列表缓存
2. 强制更新板块行情缓存

## 缓存管理

### 查看缓存状态
访问系统设置页面，查看：
- 缓存文件数量
- 缓存总大小
- 最后更新时间

### 清理缓存
在系统设置页面：
- **清理所有缓存** - 删除所有缓存文件
- **仅清理板块缓存** - 删除板块相关缓存

## 技术细节

### 缓存有效期
- `board_list_cache.json` - 30天
- `board_quote_cache.json` - 24小时（宽松策略）

### 降级策略
1. 尝试实时获取
2. 失败则使用缓存
3. 缓存过期则返回空

### 数据合并
板块热度功能会自动合并：
- 板块列表（名称、代码）
- 板块行情（涨跌幅、情绪分）

## 常见问题

**Q: 为什么板块列表缓存30天？**
A: 板块列表不常变化，长期缓存可以减少API调用，提高稳定性。

**Q: 板块行情缓存24小时会不会太久？**
A: 这是降级策略。正常情况下会实时获取，只有在API失败时才使用缓存。

**Q: 如何确保数据是最新的？**
A: 在交易时间运行 `update_board_cache.py`，强制更新所有缓存。

**Q: 缓存文件在哪里？**
A: `.cache/board_list_cache.json` 和 `.cache/board_quote_cache.json`
